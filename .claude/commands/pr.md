現在の作業ツリーの変更から Pull Request を作成してください。ユーザーからの補足: $ARGUMENTS

以下の手順に従うこと:

## 1. 変更内容の把握

- `git status` と `git diff` で変更内容を確認する
- 変更が何もない場合はその旨を伝えて終了する
- 意図しないファイル(ビルド成果物、一時ファイル等)が混ざっていないか確認する

## 2. コミット前チェック(CLAUDE.md 落とし穴 #12)

変更されたディレクトリに応じて必ず実行し、失敗したら修正してから進む:

- `backend/` に変更がある場合: `cd backend && ./gradlew check`
- `frontend/` に変更がある場合: `cd frontend && npm run check`
- ドキュメントのみの変更ならチェック不要

## 3. ドキュメント整合の確認(CLAUDE.md ドキュメント運用ルール)

- 実装と設計ドキュメント(`.claude/01_development_docs/` 等)が乖離する変更なら、同じPRでドキュメントも更新されているか確認する
- 特にマッチング・順位計算の変更は `05_swiss_pairing_algorithm.md` の更新が先行しているべき

## 4. ブランチ作成

- `main` にいる場合は必ずブランチを切る: `git checkout -b <prefix>/<英語の短いスラッグ>`
- prefix は変更内容に応じて選ぶ: `feat/` `fix/` `chore/` `docs/` `refactor/` `test/`
- 既に作業ブランチにいる場合はそのまま使う

## 5. コミット

- メッセージ形式: `<prefix>: <日本語の要約>`(1行目)+ 空行 + 必要なら本文(日本語)
- 末尾に必ず付ける: `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`
- 関連する変更は1コミットにまとめる。無関係な変更が混ざっていたらユーザーに確認する

## 6. プッシュとPR作成

- `git push -u origin <ブランチ名>`
- `gh pr create` で作成。タイトルはコミットメッセージ1行目と揃える
- PR本文は以下のテンプレート(日本語):

```markdown
## 概要

(何を・なぜ。バグ修正なら再現条件とエラー内容も)

## 原因

(バグ修正の場合のみ。根本原因の説明)

## 変更内容

- (箇条書き)

## テスト

- (実行したチェック・テストと結果)

🤖 Generated with [Claude Code](https://claude.com/claude-code)
```

## 7. 報告

- PR の URL を報告する
- CI で AIレビュー(`.github/workflows/ai-review.yml`)が自動実行されることを添え、レビュー結果を確認してからマージするよう伝える

## 8. 依存関係のある複数PR(スタックPR)を同一セッションで作成する場合

大きい機能を「ドキュメント→スキーマ→ドメイン→バックエンド→フロントエンド→E2E」のように複数PRへ分割し、各PRのbaseを前のPRのブランチにする場合(例: 団体戦機能のPR #48〜#56)、以下を必ずユーザーと共有・合意してから進める。守らないと、下流PRが軒並み「内容は同じなのにconflict」になり手戻りが発生する(実例: PR #48〜#50が`main`にsquashマージされた際、以前から積まれていた#51〜#56が連鎖的にconflict化した)。

**根本原因**: Squash and merge・Rebase and mergeはマージのたびにコミットハッシュを書き換える。スタックの下位PRがこの方式でmainに統合されると、上位の未マージPRは「元のコミット」を履歴に持ったままなので、内容が同一でもハッシュが一致せず衝突する。

**回避策(このスタックが完結するまで一貫して守る)**:

- **マージ方式は "Create a merge commit" に統一する**(Squash and merge / Rebase and mergeは使わない)。理由は上記の通りコミットハッシュを保つため。この方針はスタック作成時点でユーザーに確認し、GitHubリポジトリまたは該当PR群の設定に反映する。
- **マージ順序は必ず底(baseがmainのPR)から**。上位PRを先にマージしても、そのPRのbaseがmainでない限りmainには反映されない。
- 底のPRがmainにマージされたら、次のPRのbaseを`main`に貼り替える(`gh pr edit <番号> --base main`)。貼り替え後は共有履歴が一致しているため差分・conflictは出ないはずである。これを1段ずつ繰り返す。
- **途中のPRに指摘修正コミットが追加された場合**、下流ブランチへの取り込みは `git rebase` ではなく `git merge <更新されたbaseブランチ>` を使う。マージコミットはハッシュを書き換えないため、後続の統合時に重複コンフリクトが再発しない。rebaseは(スタック運用中に限り)避ける。

**将来的な選択肢の検討メモ(Graphite等のスタックドPRツール)**:

Graphite (`gt` CLI)のようなツールは、下位PRのマージ後に上位PRのbase retargetingとrebase/restackを自動化できる。以下を踏まえて、必要になった時点でユーザーと導入可否を相談する:

- 利点: base貼り替え・再スタックの手作業をなくせる。3段以上のスタックが頻発するなら手作業のミスを減らせる
- 懸念: 外部ツールへのGitHub App権限付与が必要、追加の学習コストとCLIツールへの依存が増える、本プロジェクトの規模(個人〜小規模チーム開発)では上記の手動ルールで十分回避できている
- 現時点の結論: 未導入。スタックPRが頻発する・関わる人数が増えるなど運用負荷が明確に上がったタイミングで再検討する
