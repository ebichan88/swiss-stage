# E2Eテスト設計書

## 1. 基本方針

- ツール: **Playwright**(`frontend/tests/e2e/`)
- 対象: **クリティカルパスのみ**(大会当日に絶対に動かなければならないフロー)
- 実行タイミング: リリース前・大会前(PRごとには実行しない)
- バックエンド + DynamoDB Local を起動した状態で実行する

---

## 2. クリティカルパス定義

> 「これが動かないと大会が止まる」フローを最優先でテストする。

### CP1: 大会運営の一気通貫(最重要)

```
ログイン → 大会作成(囲碁・5回戦) → 参加者をCSVインポート(16名)
→ 大会開始 → 第1ラウンド生成 → 全対局の結果入力 → ラウンド確定
→ 第2ラウンド生成(再戦がないこと) → 順位表表示(SOS順であること)
```

### CP2: 参加者の結果送信(スマホ)

```
共有URLにアクセス(モバイルビューポート 375x667)
→ 組み合わせ表で自分の卓を確認 → 結果入力 → 送信
→ 運営者画面に結果が反映されている
```

### CP3: 奇数人数・BYE

```
15名で大会開始 → ラウンド生成 → BYEが1件存在
→ 全ラウンドを通して同一参加者にBYEが2回付かない
```

### CP4: 異常系(大会当日に起きること)

- 未入力の対局がある状態でラウンド確定 → 警告が表示される
- 2つのブラウザで同じ対局に別の結果を入力 → 後勝ちにならず409エラー表示
- 無効化した共有トークンでアクセス → エラーページ表示

---

## 3. 実装ルール

1. **セレクタは `getByRole` / `getByLabel` を優先**。壊れやすいCSSセレクタ禁止
2. **テストデータの分離**: 各テストは自分で大会を作成し、テスト間で大会を共有しない
3. **認証のセットアップ**: Google OAuth2を実際に通さない。E2E用に `local` プロファイル限定のテストログインエンドポイント(`/api/v1/auth/test-login`)を用意する(**本番ビルドには絶対に含めない** — `@Profile("local")` 必須)
4. **CSVインポート用フィクスチャ**: `tests/e2e/fixtures/participants_16.csv` 等を用意(UTF-8版とShift_JIS版の両方)
5. **失敗時の調査**: `trace: on-first-retry` を設定し、失敗時はトレースで原因調査

---

## 4. 実行コマンド

```bash
# 前提: backend起動済み + DynamoDB Local起動済み
npm run test:e2e          # ヘッドレス実行
npm run test:e2e:ui       # UIモード(デバッグ用)
```

---

## 5. 大会前チェックリスト(手動確認)

E2E自動テストに加え、実大会の前には以下を実機で確認する:

- [ ] 実際のスマートフォン(iOS Safari / Android Chrome)で共有ページを表示・結果送信
- [ ] 会場想定の低速回線(Chrome DevToolsのSlow 3G)で結果送信 → リトライ動作
- [ ] 実際の参加者数のCSVでインポート
- [ ] CloudWatchアラームが有効になっていること
