## 2. 画面遷移図

### 運営者フロー

```
S01 トップ
 └─▶ S02 ログイン ─(Google OAuth2)─▶ S03 大会一覧
                                        ├─▶ S04 大会作成 ──▶ S05 大会管理
                                        └─▶ S05 大会管理
                                             ├─▶ S06 参加者管理(CSVインポート含む)
                                             ├─▶ S07 ラウンド管理
                                             │     ├─ 組み合わせ生成
                                             │     ├─ 結果入力
                                             │     └─ ラウンド確定 ─▶ 次ラウンドへ
                                             ├─▶ S08 順位表
                                             └─▶ S09 大会設定(公開範囲・共有URL・削除)
```

### 参加者フロー(共有URL経由・ログイン不要)

```
共有URL(QR/リンク) ─▶ S10 共有ページ
                        ├─ 組み合わせ確認(自分の卓番号・対戦相手)
                        ├─▶ S11 結果自己申告(大会設定で許可時のみ。両者の申告が一致すると自動確定)
                        ├─ 順位表確認
                        └─ 戦績一覧確認(全ラウンドの対戦相手・結果を一覧)
```

## 3. 大会運営の状態と画面の関係

| 大会状態 | S06 参加者管理 | S07 ラウンド管理 | S10 共有ページ |
|---------|--------------|----------------|--------------|
| PREPARING | 追加・編集・削除可 | ラウンド生成不可 | 「開始前」表示 |
| IN_PROGRESS | 棄権処理のみ可 | 生成・入力・確定可 | 組み合わせ・順位表示 |
| FINISHED | 閲覧のみ | 閲覧のみ | 最終結果表示 |

## 4. ナビゲーション設計

- **運営者画面**: 左サイドバー(PC) / 下部タブ(スマホ)。項目: 概要・参加者・ラウンド・順位・設定
- **共有ページ**: タブ切り替え(組み合わせ / 順位表 / 戦績一覧)。ヘッダーに大会名と現在ラウンドを常時表示
- 大会管理系画面(S05〜S09)は共通レイアウト(`TournamentLayout`)でラップし、大会名・状態バッジを常時表示する

---

## 5. UX上の重要ルール

1. **結果入力は「自分はどちらか」選択 + 「勝ち/負け」選択 + 確認の3ステップ**: 対戦相手と結果を偽って一方的に確定できないよう、
   両者がそれぞれ自分の結果を申告する(片方の申告だけでは確定しない。一致すると自動確定)
2. **破壊的操作は必ず確認**: ラウンド確定・大会削除・トークン再発行はダイアログで確認。大会削除は大会名の入力を要求
3. **ラウンド確定のブロックは「一切未着手」の対局のみ**: 運営者・参加者のいずれも触れていない対局が残る場合は確定できない。
   片方のみ申告・申告が一致しない対局は警告表示のみでブロックしない(運営者の裁量で確定できる。詳細は
   `05_swiss_pairing_algorithm.md` の対象外・結果確定の運用ルールとして `13_security_design.md` 参照)
4. **オフライン考慮**: 結果送信失敗時はリトライ可能なエラー表示(会場の電波状況が悪いケースを想定)
5. **URLで状態復元可能**: すべての画面はURL直アクセスで再現できること(SPAでもディープリンクを壊さない)
