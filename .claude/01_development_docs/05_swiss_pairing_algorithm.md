# スイス方式マッチング・順位計算アルゴリズム設計書

> **このドキュメントはSwiss Stageの心臓部である。**
> ここに書かれたルールは domain 層の実装・テストと常に一致していなければならない。
> 仕様変更時は必ず本ドキュメントを先に更新すること。

## 1. 用語定義

| 用語 | 定義 |
|------|------|
| 棋力 | 段級位。29段階の順序付き値(§1.1)。初回ラウンドのペアリング基準 |
| 勝点 | 勝ち=1、負け=0、引き分け=0.5、不戦勝(BYE)=1 |
| スコアグループ | 同一勝点の参加者の集合 |
| グループ | 棋力帯などで大会内を区分するクラス分け(例: Aグループ=五段以上)。全大会が常に1つ以上のグループを持ち、マッチング・順位計算はグループごとに独立して行う(§2.4)。**スコアグループとは別概念** |
| フロート | 自分のスコアグループ外の相手と対戦すること(下方フロート=下位グループと対戦) |
| SOS | Sum of Opponents' Scores。対戦した相手の勝点合計(不戦勝の相手は除外) |
| SOSOS | 対戦した相手のSOSの合計 |
| BYE(不戦勝) | 奇数人数時に対戦相手がつかないこと。勝点1を得る |

### 1.1 棋力(段級位)の定義

囲碁・将棋共通で、以下の**29段階**の順序付き列挙とする(`Rank` enum、定義は `schema/openapi.yaml`):

```
弱 ── 20級 < 19級 < … < 2級 < 1級 < 初段 < 2段 < … < 9段 ── 強
```

- 一番弱い棋力は **20級**、一番強い棋力は **9段**
- **1級の次は初段**(「1段」は存在しない)
- 強さの比較は enum の宣言順ではなく、明示的な `sortOrder`(段=負、級=正、**小さいほど強い**: 9段=-9 … 初段=-1、1級=1 … 20級=20)で行う
- 棋力は任意入力(null許容)。ソート時、未入力者は20級よりさらに弱い扱い(末尾)とする
- 同棋力(および未入力同士)の順序はエントリー順(`entryOrder`)で決める

## 2. マッチングアルゴリズム

### 2.1 初回ラウンド

**できるだけ棋力の近い者同士を対戦させる。**

1. 棋力の強い順にソートする(§1.1。同棋力・未入力はエントリー順、未入力者は末尾)
2. 隣接ペアリング: 先頭から2人ずつペアにする(1番目vs2番目、3番目vs4番目…)
   - 大会設定で「ランダムペアリング」も選択可能(棋力を無視して無作為に組む)
3. 奇数人数の場合、ソート末尾(最弱)の参加者にBYEを与える

### 2.2 第2ラウンド以降

1. 参加者(ACTIVE のみ)を勝点降順でスコアグループに分ける
2. 勝点の高いグループから順にグループ内でペアリングする
3. グループ内が奇数の場合、グループ最下位(SOS最小)を下位グループへフロートさせる
4. ペア決定の制約:
   - **[絶対] 再戦禁止**: 過去に対戦したペアは組まない
   - **[絶対] BYE重複禁止**: 一度BYEを受けた参加者には再度BYEを与えない
   - **[優先] 同一所属回避**: 同じ `organization` 同士の対戦を可能な限り避ける(大会設定でON/OFF可、デフォルトON)
   - **[優先] フロート回数の平準化**: 同じ参加者が連続フロートしないようにする
5. 制約を満たすペアが見つからない場合はバックトラックし、それでも不可能な場合のみ絶対制約以外を緩和する
   - 緩和順序: 同一所属回避 → フロート平準化
   - **再戦禁止とBYE重複禁止はどうしても解がない場合のみ緩和し、その際はUIに警告を表示する**
6. 奇数人数の場合のBYE付与: 「BYE未経験」かつ「最下位スコアグループ」かつ「SOS最小」の参加者

### 2.3 途中棄権(WITHDRAWN)の扱い

- 棄権者は以降のマッチング対象から除外する
- 棄権者の過去の対局結果は順位計算に残す
- 対局中に棄権が発生した場合、その対局は相手の不戦勝(BYEではなく `PLAYER_X_WIN` + 備考)とする

### 2.4 グループ大会のマッチング

**全大会はグループ(棋力帯クラス分け)を常に1つ以上持ち、マッチングはグループごとに独立して行う。**
大会作成時にデフォルトグループ「A」を自動作成する。グループ分けしない大会はグループ1つのまま運用する(単一グループ=従来の「グループなし大会」と同じ挙動)。
参加者・対局は常にいずれか1つのグループに帰属する(**groupId 必須。「未割当」「グループなし」状態は存在しない**)。

- 各グループ内で §2.1〜2.3 のルールをそのまま適用する。再戦禁止・BYE重複禁止・同一所属回避・SOS などの判定はすべて**グループ内で完結**する(グループをまたぐ対戦は発生しない)
- **ラウンド進行は大会全体で同期**する。「第Nラウンド生成」で全グループ分の組み合わせを一括生成し、確定も大会単位で行う。グループごとに異なるラウンド番号を進めることはない
- **BYEはグループ内で発生**する: グループの ACTIVE 人数が奇数なら、そのグループ内で §2.1/§2.2 のBYE付与ルールを適用する
- ACTIVE が1名のみのグループは、その参加者にBYEを直接付与する(BYE経験済みなら `BYE_REPEAT` を relaxations に記録して重複付与する)。ACTIVE が0名のグループはスキップする
- `relaxations` は**全グループの和集合**を返す(どのグループで緩和が起きても大会単位の警告として表示する)
- **卓番号はグループごとに1始まり**で採番する。表示は「{グループ名}-{卓番号}」形式(例: `A-1`)。**グループが1つだけの大会は表示上グループ見出し・卓番号プレフィックスを省略する**(データ上は常にグループに帰属する。省略は表示層のみの規則)

#### グループの制約

- グループの定義(作成・改名・削除)と参加者の割当変更は **PREPARING 中のみ**可能
- グループは常に1つ以上存在する(**最後の1グループは削除できない**)
- グループを削除すると、割当済みの参加者は**直前のグループ(先頭グループの削除時は直後のグループ)**へ移動する
- 参加者の追加(個別・CSV)でグループ指定を省略した場合は**先頭のグループ(定義順)**に割り当てる
- 大会開始時に以下を検証する(違反時は開始不可):
  - 全 ACTIVE 参加者の割当先が定義済みグループであること
  - 各グループの ACTIVE 参加者が2名以上であること(参加者0名の空グループも開始不可。不要なグループは削除してから開始する)

#### 段級位による自動振り分け

運営者の一括操作「段級位で自動振り分け」は、以下の決定的なルールで全 ACTIVE 参加者の割当案を適用する(適用後に個別調整可能):

1. 参加者を棋力の強い順にソートする(§1.1。同棋力・未入力はエントリー順、未入力者は末尾)
2. **グループ定義順(作成順)に、できるだけ均等な人数で先頭から分割**する
   - 端数は先頭側のグループに1名ずつ多く割り当てる(例: 47名を3グループ → 16/16/15)
   - したがってグループは**強い帯から順に定義する**運用を前提とする(A=最強帯)
3. 棄権(WITHDRAWN)中の参加者は自動振り分けの対象外とし、**現在の割当を維持**する

## 3. 順位計算

### 3.1 順位決定基準(上から順に適用)

| 優先 | 基準 | 備考 |
|------|------|------|
| 1 | 勝点(多い順) | |
| 2 | SOS(多い順) | 不戦勝・不戦敗の仮想相手は除外 |
| 3 | SOSOS(多い順) | |
| 4 | 直接対決の結果 | 該当2名が対戦済みの場合のみ |
| 5 | エントリー順(小さい順) | 最終手段。同順位を許す設定も可(デフォルト: 同順位表示) |

### 3.2 SOS計算の詳細ルール

- BYEを受けたラウンドは「相手なし」としてSOS計算から除外する
  (代替として「自身の勝点を加算する」流派もあるが、**本プロダクトでは除外方式を採用**)
- 棄権者との対局は実際の結果と棄権者の最終勝点をそのまま使う
- 引き分け(囲碁の持碁等)は0.5勝として扱う

### 3.3 グループ大会の順位

- **順位は常にグループごとに独立して計算**する(順位はグループ内で 1..n。§2.4 のとおり全大会がグループを持つ)
- SOS/SOSOS もグループ内の対局のみから計算される(§2.4 によりグループ外との対局は存在しない)
- 順位表APIはグループごとの順位リストを返す(`GroupStandings[]`。`group` は常に非null。定義は `schema/openapi.yaml`)

### 3.4 計算タイミング

- 順位表APIが呼ばれるたびに全対局結果から再計算する(結果修正に自動追従)
- 実装は `domain/service/StandingCalculator` に置き、**入力(参加者+全対局) → 出力(順位リスト)の純粋関数**とする

## 4. テスト要件(必須ケース)

マッチング・順位計算のテストは以下を必ずカバーする(`09_test_strategy.md` 参照):

1. 偶数/奇数人数(BYE発生)
2. 初回ラウンドが棋力順の隣接ペアになること(1級と初段の境界、棋力未入力者の末尾扱いを含む)
3. 再戦禁止が効いていること(全ラウンド通して重複ペアなし)
4. BYEが同一参加者に2回付かないこと
5. 同一所属回避が働くこと・解がない場合に緩和されること
6. 途中棄権後のマッチング・順位計算
7. SOS/SOSOSのタイブレーク(BYE除外の検証含む)
8. 16人5回戦・300人8回戦のプロパティテスト(制約違反ゼロを機械的に検証)
9. 結果未入力ラウンドがある状態での順位表(暫定順位)
10. グループ大会: グループごとの独立ペアリング(グループ横断ペアなし)・グループ内BYE・卓番号グループ内1始まり・relaxations の和集合
11. グループ大会: 順位のグループ独立計算、1名グループへのBYE直付与(BYE済なら `BYE_REPEAT`)、0名グループのスキップ
12. 自動振り分けの均等分割・端数処理・棋力未入力者の末尾扱い・棄権者の割当維持、開始時検証(2名未満グループ・空グループで開始不可)
13. グループが1つだけの大会が従来の「グループなし大会」と同一の挙動であること(単一母集団でのマッチング・順位計算・大会通し卓番号)
14. 団体戦: エントリー順のみでの初回ペアリング(棋力シート・同一所属回避が適用されないこと)、再戦禁止・BYE重複禁止がチーム単位で効くこと
15. 団体戦: ボード結果からのチーム点数集計(勝/負/分/両者負けの混在)・同点判定、SOS/SOSOSがチーム単位で個人戦と同じ計算方法になること
16. 団体戦: 必須boardPositionが未充足のチームがある場合の大会開始拒否、補欠人数上限超過の拒否
17. 団体戦: ボード単位の自己申告整合判定(一部ボードのみ不一致でも他ボードは確定すること)

## 5. 団体戦(3人制/5人制)

団体戦は「チーム」をマッチングの単位とし、対局(TeamMatch)は主将戦・副将戦・…という**ボード位置単位**の結果の集合として扱う。§1〜4の個人戦ルールとの差分のみを以下に定義する。マッチング(バックトラッキング・制約緩和)とSOS/SOSOS計算の核心ロジックは個人戦と共通のエンジンを使い、以下のアダプタ差分のみで動作を変える(個人戦・団体戦でロジックを二重実装しない)。

### 5.1 チーム編成

- 3人制: 主将・副将・三将(boardPosition=1..3、全員必須)+補欠最大2名(boardPosition=null)
- 5人制: 主将・副将・三将・四将・五将(boardPosition=1..5、全員必須)+補欠最大3名(boardPosition=null)
- 補欠が実際にどのボードで交代出場したかは記録・追跡しない(オーダー管理はしない)。対局結果は氏名ではなく**ボード位置(役割)**に紐づく
- 大会開始には、全ACTIVEチームの必須boardPosition(1..teamSize)が過不足なく1名ずつ埋まっていることを要する(未充足チームがあれば開始不可)

### 5.2 マッチング(初回ラウンド・2ラウンド以降)

- **初回ラウンドのシードなし**: チームには個人戦の棋力に相当する単一の強さの指標がないため、§2.1の棋力順ソートは適用しない。**エントリー順のみ**でペアリングする(隣接ペアリングの基準キーがすべて等しい状態と同義)
- **同一所属回避なし**: チームには「所属」の概念がないため、§2.2の「同一所属回避」制約は適用しない(絶対制約である再戦禁止・BYE重複禁止のみを適用する)
- それ以外(再戦禁止・BYE重複禁止・フロート平準化・バックトラック・グループ独立マッチング・棄権時の扱い)は§2.1〜2.4と同一のルールをチーム単位で適用する
- グループ(棋力帯クラス分け)機能は団体戦でもチーム単位で利用できるが、「段級位による自動振り分け」(§2.4)はチームに棋力がないため団体戦では提供しない(グループ割当は手動のみ)

### 5.3 対局結果とチーム全体の勝敗判定

- 各ボードの結果は個人戦と同じ結果種別(勝ち/負け/引き分け/両者負け)を選択できる
- チーム全体の勝敗は**単一のenumとして保存せず**、各ボードの結果を個人戦と同じ点数変換(勝=1・負=0・分=0.5・両者負け=0)でチーム視点の点数に変換し、その合計(`team1Points`/`team2Points`)を比較して都度導出する。同点(引き分け相当)も起こり得る前提で表示・順位計算する
- ラウンド確定のブロック判定は個人戦の「対局が一切未着手」と同じ粒度: TeamMatchの全ボードの結果および両者の申告が全てNONEの場合のみ未着手とみなす。1ボードでも入力・申告があれば警告のみで確定可能(個人戦の運用ルールを踏襲)

### 5.4 共有トークン経由の自己申告

- 参加者はまず「自チームがteam1/team2のどちらか」を選択し、全ボード分の結果を1回の送信でまとめて申告する
- 確定はボード単位で独立に行う: 両者の申告がボードごとに一致した時点でそのボードのみ確定する(他のボードの確定状況に影響しない)。個人戦の自己申告の突き合わせ運用(`13_security_design.md`)をボード粒度に適用したものである
- 不一致のボードは、確定させず「誰が何を申告したか」を運営者画面・共有画面の両方に具体的に表示する

### 5.5 順位計算

- チームの勝点・SOS・SOSOS・直接対決・エントリー順によるタイブレークは§3と全く同じ基準・計算方法をチーム単位に適用する(点数の元になる「対局結果」がボード集計値である点のみが個人戦と異なる)
- 戦績一覧・組み合わせ・順位表にはチーム名のみを表示し、メンバーの個人名は一切表示しない(補欠交代の可能性・個人情報保護のため)

## 6. 非対応事項(MVP)

- ハンデ戦(囲碁の置き碁)の考慮 → 将来検討
- 手番(先手/後手・黒/白)の平準化 → 将来検討(現状は卓番号のみ割り当て)
- マクマホン方式・リーグ戦 → 対象外
- 大会開始後のグループ間移動・グループごとの独立したラウンド進行 → 対象外(§2.4 のとおり進行は大会全体で同期)
- 団体戦のオーダー管理(補欠が誰と交代したか)・対局結果への個人名の紐付け → 対象外(§5.1)
- 団体戦の初回ラウンドにおける棋力ベースのシード・チームの段級位による自動グループ振り分け → 対象外(§5.2)
