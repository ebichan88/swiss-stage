---

## 1. 我々はなぜここにいるのか

囲碁・将棋大会の運営は、いまだに紙と手作業に支えられている。

- 紙の対局カードへの記入・提出
- 手作業での結果転記・順位集計
- 手作業でのスイス方式組み合わせ

Google Spreadsheet + GAS による試作で「スマホから対局結果を送信できる」体験が高く評価された実績があり、
**大会運営全体をデジタル化する価値**が確認できている。

その価値を、誰でも使えるプロダクトとして届けるためにこのプロジェクトは存在する。

---

## 2. エレベーターピッチ

**囲碁・将棋大会を運営するすべての人**向けの、
**大会運営プラットフォーム「Swiss Stage」**は、
**スイス方式マッチング・結果集計・順位表示を自動化するWebサービス**である。

これは**紙と手作業による運営から運営者を解放し**、
**Google Spreadsheet や汎用ツールの寄せ集めとは違って**、
**囲碁・将棋大会に特化した一貫した運営体験**を提供する。

---

## 3. やらないことリスト(MVP)

| やること | やらないこと | あとで決めること |
|---------|------------|----------------|
| 個人戦のスイス方式大会 | 団体戦 | 年間ランキング |
| Googleログイン(運営者) | エントリーサイト | QRコード受付 |
| 大会作成・参加者管理 | 決済 | 棋譜管理 |
| CSVインポート | ネイティブアプリ | API公開 |
| マッチング・結果入力・順位表示 | 囲碁・将棋以外の競技 | 参加者プロフィール |
| 参照URLによる大会共有 | リアルタイム同時編集 | 通算戦績 |

---

## 4. プロジェクトコミュニティ

- **開発者**: 個人開発(1名) + Claude Code
- **最初のユーザー**: 開発者自身が運営に関わる実業団囲碁大会
- **将来のユーザー**: 企業・サークルの大会運営者、大会参加者

---

## 5. 技術的な解決策の概要

- フロントエンド: React 18 + TypeScript + Vite + Material-UI (SPA)
- バックエンド: Java 21 + Spring Boot 3.x (REST API)
- データベース: DynamoDB (シングルテーブル設計)
- インフラ: AWS (EC2 + ALB + Route53 + CloudWatch)
- 認証: Google OAuth2 (運営者) / 参照URLトークン (参加者)

詳細は `.claude/01_development_docs/01_architecture_design.md` を参照。

## 6. 夜も眠れなくなる問題

| リスク | 対策 |
|-------|------|
| マッチング・順位計算のバグ(大会当日に発覚すると致命的) | domain層90%カバレッジ + スイス方式アルゴリズムの徹底テスト(`05_swiss_pairing_algorithm.md`) |
| 大会当日のサーバーダウン | CloudWatch監視 + 手動運用へのフォールバック手順を用意 |
| 個人情報(氏名・企業名)の漏洩 | 最小限の情報のみ保持、公開範囲設定、認可の徹底(`13_security_design.md`) |
| 大会当日の同時アクセス集中(300名規模) | 順位表のキャッシュ、DynamoDBオンデマンド(`14_performance_optimization.md`) |
| 個人開発のモチベーション枯渇 | MVPを小さく保ち、自分の大会で使うことを最優先にする |

---

## 7. 期間を見極める

- **フェーズ1(MVP)**: 大会作成 → 参加者管理 → マッチング → 結果入力 → 順位表示 の一気通貫
- **フェーズ2**: 実大会での試験運用・フィードバック反映
- **フェーズ3**: 将来構想機能(団体戦、エントリー等)の段階的追加
